# Agent 面试

## 一、基础概念

### 1. 什么是 AI Agent？

**考察点：** 候选人是否理解 Agent 与普通大模型调用的区别。

**参考回答：**

AI Agent 是一种能够基于目标自主进行任务规划、工具调用、记忆管理和结果反馈的智能系统。它不仅能回答问题，还可以根据上下文拆解任务、选择工具、执行操作，并根据执行结果进行迭代。

**工作流**是一套预先设计好的流程。它通常包含：
- 固定步骤
- 明确的输入和输出
- 条件判断
- 节点之间的顺序关系
- 人工或系统触发

**Agent** 可以理解为一个具备一定"自主能力"的智能执行者。它通常可以：
- 理解目标
- 拆解任务
- 自主选择工具
- 根据中间结果调整计划
- 多轮思考和执行
- 在不确定场景中做决策

### 2. Agent 和传统 Chatbot 有什么区别？

**考察点：** 是否理解 Agent 的自主性和执行能力。

**参考回答：**

传统 Chatbot 主要是问答式交互，通常只根据用户输入生成回复；而 Agent 更强调目标驱动，具备任务规划、工具调用、状态管理、长期记忆和多轮执行能力。

### 3. Agent 系统通常由哪些核心模块组成？

**考察点：** 系统架构理解能力。

**参考回答：**

常见模块包括：
1. 大语言模型 LLM
2. Prompt 管理模块
3. 任务规划 Planner
4. 工具调用 Tool Use
5. 记忆系统 Memory
6. 执行器 Executor
7. 观察与反馈 Observation
8. 安全与权限控制模块
9. 日志与监控模块

## 二、Prompt Engineering

### 1. 如何设计一个稳定的 Agent Prompt？

**考察点：** Prompt 设计能力。

**参考回答：**

一个稳定的 Agent Prompt 通常应包含：
1. 角色定义
2. 任务目标
3. 可使用工具说明
4. 输出格式约束
5. 行为边界
6. 错误处理策略
7. 示例输入输出

### 2. System Prompt、User Prompt 和 Tool Prompt 分别有什么作用？

**考察点：** Prompt 分层理解。

**参考回答：**

System Prompt 用于定义模型的最高优先级行为规则；User Prompt 表示用户当前任务需求；Tool Prompt 用于描述工具的功能、参数和调用方式，帮助模型正确使用外部能力。

## 三、工具调用与 Function Calling

### 1. 什么是 Function Calling？

**考察点：** 对工具调用机制的理解。

**参考回答：**

Function Calling 是指大模型根据用户意图选择预定义函数，并生成结构化参数，由外部系统执行函数后再将结果返回给模型，模型基于结果继续推理或生成最终答案。

### 2. 如何避免 Agent 错误调用工具？

**考察点：** 工程稳定性与安全意识。

**参考回答：**

可以通过以下方式降低错误调用：
1. 明确工具描述和适用场景
2. 使用 JSON Schema 限制参数格式
3. 增加参数校验
4. 对高风险操作增加人工确认
5. 设置工具调用权限
6. 记录工具调用日志
7. 对异常结果进行重试或降级处理

### 3. 如果模型生成的工具参数不合法，应该如何处理？

**考察点：** 异常处理能力。

**参考回答：**

应先进行参数校验，如果不合法，可以让模型重新生成参数，或使用默认值、兜底逻辑、错误提示等方式处理。同时应记录异常日志，便于后续优化 Prompt 或工具描述。

## 四、Agent 规划能力

### 1. ReAct 框架是什么？

**考察点：** 经典 Agent 框架理解。

**参考回答：**

ReAct 是 Reasoning and Acting 的缩写，强调模型在推理和行动之间循环。模型先进行思考（Thought），决定是否调用工具（Action），观察工具结果（Observation）后继续推理，直到完成任务。

传统大语言模型通常擅长生成文本，但在复杂任务中容易出现两个问题：
1. **只会"想"，不会"做"**  
   例如它可以告诉你"应该去查天气"，但自己不能真正查询天气。
2. **容易凭空编造**  
   如果模型不知道最新信息，可能会直接生成一个看似合理但错误的答案。

ReAct 框架通过让模型在推理过程中调用外部工具，例如搜索引擎、数据库、计算器、API 等，来减少这类问题。

**示例：**
```
Question: 北京今天适合穿什么？
Thought: 我需要知道北京今天的天气。
Action: 查询天气工具，北京
Observation: 北京今天晴，最高温18℃，最低温6℃。
Thought: 天气较凉，昼夜温差较大，需要建议用户穿外套。
Answer: 北京今天适合穿长袖、薄外套，早晚可以加一件稍厚外套。
```

### 2. Plan-and-Execute 模式是什么？

**考察点：** 任务拆解能力理解。

**参考回答：**

Plan-and-Execute 是一种先规划后执行的 Agent 模式。Agent 首先将复杂任务拆解为多个步骤，然后按步骤执行，每一步可调用工具并根据结果调整后续计划。

**1）Planner：规划器**

负责理解用户目标，并把复杂任务拆解成多个步骤。

例如用户说："帮我写一份新能源汽车行业分析报告。"

Planner 可能会拆成：
1. 明确报告结构
2. 收集新能源汽车市场规模数据
3. 分析主要企业
4. 分析政策和技术趋势
5. 总结机会与风险
6. 生成完整报告

**2）Executor：执行器**

负责按照 Planner 给出的步骤逐项执行。例如：
- 查询资料
- 调用搜索工具
- 分析数据
- 生成文本
- 调用数据库
- 输出结果

**ReAct vs Plan-and-Execute 对比：**

| 对比项 | ReAct | Plan-and-Execute |
|--------|-------|------------------|
| 核心方式 | 边推理边行动 | 先规划再执行 |
| 决策节奏 | 每一步都动态判断 | 先整体拆解任务 |
| 适合任务 | 信息不确定、需要频繁反馈 | 复杂但可拆解的任务 |
| 计划结构 | 通常较隐式、局部 | 通常较显式、全局 |
| 灵活性 | 高 | 中高 |
| 可控性 | 相对较弱 | 相对更强 |
| 成本 | 可能多轮循环较多 | 计划清晰时更稳定 |
| 典型场景 | 搜索问答、排障、工具调用 | 报告生成、项目任务、复杂流程 |

### 3. 如何处理 Agent 执行过程中的死循环问题？

**考察点：** Agent 运行控制能力。

**参考回答：**

Agent 在执行任务时，可能会因为目标不清晰、工具调用失败、上下文误判、状态未更新等原因反复执行同一动作，形成死循环。常见处理方式如下：

**1. 设置最大执行步数**

为 Agent 设置最大推理轮次或最大工具调用次数。这样可以防止 Agent 无限运行，避免资源浪费。

**2. 检测重复行为**

记录 Agent 的历史操作轨迹，包括：
- 调用过的工具
- 输入参数
- 工具返回结果
- Agent 的中间决策
- 当前任务状态

如果发现 Agent 连续多次执行相同操作，且结果没有变化，就可以判定为疑似死循环。

**3. 引入状态机机制**

通过状态机管理 Agent 的执行流程，避免 Agent 在同一状态中反复停留。

```
任务开始 → 信息收集 → 工具调用 → 结果验证 → 总结输出 → 结束
```

每个状态都需要明确：
- 进入条件
- 退出条件
- 最大停留次数
- 失败处理方式

如果某个状态无法继续推进，应进入异常处理流程，而不是继续重复执行。

**4. 设置超时时间**

除了限制执行步数，还应设置时间维度的限制。
- 单次任务最长执行 60 秒
- 单个工具调用最长等待 10 秒
- Agent 总运行时间超过阈值后自动终止

这样可以避免由于工具阻塞、接口无响应或推理卡住导致的长时间运行。

**5. 明确任务完成条件**

- 什么情况下任务算完成
- 什么情况下任务失败
- 什么情况下需要向用户追问
- 什么情况下应停止执行并总结已有信息

**6. 对失败结果进行分级处理**

如果工具调用失败，不应无限重试，而应根据失败类型处理。

| 失败类型 | 处理方式 |
|---------|---------|
| 网络超时 | 最多重试 1-2 次 |
| 参数错误 | 修正参数后重试一次 |
| 权限不足 | 停止执行并提示权限问题 |
| 数据不存在 | 停止搜索并返回未找到 |
| 多次无结果 | 总结当前信息并结束 |

**7. 引入反思与自检机制**

在 Agent 每执行若干步后，可以触发一次自检：
- 我是否正在重复同样的动作？
- 当前操作是否让任务更接近完成？
- 是否需要换一种策略？
- 是否应该停止并向用户说明情况？

如果自检发现没有进展，就应终止当前循环或切换策略。

**8. 使用人工介入机制**

当 Agent 多次尝试仍无法完成任务时，可以请求用户补充信息，而不是继续自动尝试。这种方式适合处理目标模糊、依赖外部信息或工具结果不稳定的任务。

**简要总结**

处理 Agent 死循环问题，核心思路是：
1. **限制次数**：设置最大步数、最大重试次数
2. **识别重复**：检测相同工具、相同参数、相同结果
3. **管理状态**：用状态机控制执行流程
4. **明确终止条件**：知道什么时候该结束
5. **失败降级**：工具失败时不要无限重试
6. **人工介入**：无法推进时请求用户补充信息

可以概括为一句话：**Agent 需要具备"继续执行"的能力，也必须具备"及时停止"的能力。**

## 五、记忆系统 Memory

### 1. Agent 的短期记忆和长期记忆有什么区别？

**考察点：** Memory 架构理解。

**参考回答：**

短期记忆通常指当前会话上下文，用于维持当前任务状态；长期记忆用于保存跨会话的重要信息，例如用户偏好、历史任务、业务知识等，通常需要结合向量数据库或结构化数据库实现。

| 对比项 | 短期记忆 | 长期记忆 |
|--------|---------|---------|
| 保存时长 | 临时保存，通常只在当前对话或当前任务中有效 | 持久保存，可跨会话、跨任务使用 |
| 信息内容 | 当前上下文、最近几轮对话、临时变量、任务进度 | 用户偏好、历史经验、知识沉淀、长期目标、重要事实 |
| 作用 | 帮助 Agent 理解当前问题、保持上下文连续 | 帮助 Agent 形成个性化理解、积累经验、提升长期表现 |
| 存储位置 | 通常在上下文窗口、运行状态或临时缓存中 | 通常在数据库、向量库、知识库或持久化存储中 |
| 更新频率 | 高频更新，随对话推进不断变化 | 低频但稳定更新，通常需要筛选、总结后写入 |
| 遗忘方式 | 对话结束或上下文超限后可能丢失 | 除非主动删除、过期或更新，一般会保留 |

简单来说：
- **短期记忆**像人的"工作记忆"，用于处理眼前任务。例如：用户刚刚说"把上面那段话改成正式风格"，Agent 需要记得"上面那段话"是什么。
- **长期记忆**像人的"长期档案"或"经验库"，用于长期个性化和知识积累。例如：Agent 记住某个用户长期偏好"回答要简洁、使用表格、偏技术风格"。

### 2. 如何实现 Agent 的长期记忆？

**考察点：** 工程落地能力。

**参考回答：**

可以将重要信息抽取后存入数据库或向量数据库。常见流程包括：
1. 信息抽取
2. 重要性判断
3. 数据存储
4. 相似度检索
5. 上下文注入
6. 记忆更新与删除

### 3. 长期记忆可能带来哪些问题？

**考察点：** 安全与隐私意识。

**参考回答：**

可能带来隐私泄露、错误记忆、过期信息干扰、数据污染等问题。因此需要设计权限控制、用户授权、记忆更新、记忆删除和敏感信息过滤机制。

## 六、RAG 与知识库

### 1. RAG 是什么？

**考察点：** 知识增强生成理解。

**参考回答：**

RAG，即 Retrieval-Augmented Generation，是检索增强生成技术。它通过从外部知识库检索相关内容，并将其作为上下文提供给大模型，从而提升回答的准确性和可追溯性。

**RAG 的基本流程：**

通常包括三步：
1. **检索 Retrieval**：用户提出问题后，系统先去知识库、文档库、数据库、网页或向量数据库中查找相关内容。
2. **增强 Augmentation**：把检索到的相关资料作为上下文，补充给大语言模型。
3. **生成 Generation**：大语言模型根据用户问题和检索到的资料，生成回答。

例如用户问："我们公司报销制度里，差旅住宿标准是多少？"

普通大模型可能不知道你公司的内部制度。RAG 系统会先检索公司制度文档，然后把相关段落提供给模型，模型再基于这些资料回答。

**为什么需要 RAG？**

大语言模型本身有几个限制：
1. 知识可能过时（模型训练完成后，对最新信息不了解）
2. 不了解私有数据（比如企业内部文档、合同、客服知识库、产品手册等）
3. 容易"幻觉"（模型可能生成看似合理但实际错误的内容）

RAG 通过引入外部资料，可以让回答更准确、更可追溯。

**RAG 的典型架构：**

一个常见的 RAG 系统包括：
- 文档数据源：PDF、Word、网页、数据库、Markdown、企业知识库等
- 文本切分：把长文档切成小片段
- 向量化 Embedding：把文本转成向量
- 向量数据库：存储和检索向量，如 Milvus、FAISS、Pinecone、Weaviate、Chroma 等
- 检索器 Retriever：根据问题找到相关片段
- 大语言模型 LLM：根据检索结果生成答案
- 重排序 Reranker：可选，用来提升检索质量

**RAG 和微调的区别：**

| 对比项 | RAG | 微调 Fine-tuning |
|--------|-----|------------------|
| 目的 | 引入外部知识 | 改变模型行为或能力 |
| 数据更新 | 更新知识库即可 | 通常需要重新训练 |
| 成本 | 相对较低 | 相对较高 |
| 可解释性 | 可以引用来源 | 不一定能追溯来源 |
| 适用场景 | 企业知识问答、客服、文档搜索 | 风格定制、格式学习、领域任务优化 |

### 2. Agent 和 RAG 有什么关系？

**考察点：** 系统组合能力。

**参考回答：**

RAG 可以作为 Agent 的一个工具或能力模块。当 Agent 需要获取外部知识时，可以调用 RAG 检索知识库，再基于检索结果进行推理和回答。

### 3. 如何提升 RAG 的检索效果？

**考察点：** RAG 工程优化经验。

**参考回答：**

常见优化方式包括：
1. 优化文档切分策略
2. 使用高质量 Embedding 模型
3. 引入 Query Rewrite
4. 使用 Hybrid Search
5. 增加 Rerank 模型
6. 控制召回数量
7. 增加元数据过滤
8. 优化知识库清洗质量

## 九、评估与优化

### 1. 如何评估一个 Agent 的效果？

**考察点：** 评估体系建设能力。

**参考回答：**

可以从以下维度评估：
1. 任务完成率
2. 回答准确率
3. 工具调用成功率
4. 平均响应时间
5. 用户满意度
6. 成本消耗
7. 安全合规率
8. 稳定性和失败率

### 2. Agent 回答不稳定应该如何优化？

**考察点：** 问题排查能力。

**参考回答：**

可以从以下方面排查：
1. Prompt 是否清晰
2. 上下文是否过长
3. 工具描述是否准确
4. 检索结果是否相关
5. 模型参数是否合理
6. 是否缺少输出格式约束
7. 是否需要增加评估集

### 1. 请设计一个简单的 Agent 执行流程伪代码。

```python
def agent_run(user_id, user_question):
    """企业内部知识问答 Agent 执行流程"""
    
    # 1. 用户身份认证
    user_info = authenticate(user_id)
    if not user_info:
        return "用户身份认证失败，请重新登录。"
    
    # 2. 权限校验
    permission_scope = get_user_permissions(user_info)
    if not permission_scope:
        return "当前用户暂无可访问的知识权限。"
    
    # 3. 理解用户问题
    intent = understand_intent(user_question)
    
    # 4. 根据意图选择处理方式
    if intent == "knowledge_qa":
        # 4.1 调用 RAG 知识检索
        documents = retrieve_knowledge(
            query=user_question,
            permission_scope=permission_scope
        )
        # 4.2 判断是否检索到相关知识
        if not documents:
            return "未在当前权限范围内找到相关知识。"
        # 4.3 生成回答
        answer = generate_answer(
            question=user_question,
            context=documents
        )
    
    elif intent == "business_query":
        # 4.4 选择合适的业务工具
        tool = select_tool(user_question)
        # 4.5 校验用户是否有工具调用权限
        if not check_tool_permission(user_info, tool):
            return "您没有权限访问该业务系统。"
        # 4.6 调用业务系统 API
        tool_result = call_tool(
            tool=tool,
            query=user_question,
            user_info=user_info
        )
        # 4.7 基于工具结果生成回答
        answer = generate_answer(
            question=user_question,
            context=tool_result
        )
    
    elif intent == "task_operation":
        # 4.8 任务类操作，例如提交工单、创建日程等
        task_plan = plan_task(user_question)
        # 4.9 执行任务前进行安全确认
        if need_user_confirm(task_plan):
            return ask_user_confirm(task_plan)
        # 4.10 执行任务
        task_result = execute_task(task_plan, user_info)
        # 4.11 生成任务执行结果
        answer = generate_answer(
            question=user_question,
            context=task_result
        )
    
    else:
        # 4.12 无法识别意图时的兜底处理
        answer = "抱歉，我暂时无法理解您的问题，请换一种方式描述。"
    
    # 5. 安全过滤
    safe_answer = safety_filter(answer)
    
    # 6. 记录日志
    record_log(
        user_id=user_id,
        question=user_question,
        intent=intent,
        answer=safe_answer
    )
    
    # 7. 返回最终结果
    return safe_answer
```

# Agent Router 面试题

## 一、基础概念题

### 1. 什么是 Agent Router？

**参考答案：**

Agent Router 是一种用于在多个 Agent、工具、模型或任务处理链路之间进行智能分发的组件。它根据用户输入、任务类型、上下文信息、系统状态等因素，决定请求应该被路由到哪个 Agent 或处理模块。

### 2. Agent Router 的核心作用是什么？

**参考答案：**

Agent Router 的核心作用包括：
1. 识别用户意图
2. 判断任务类型
3. 选择合适的 Agent 或工具
4. 管理多 Agent 协作流程
5. 提升系统响应准确率和执行效率
6. 降低单一 Agent 的复杂度

### 3. Agent Router 和普通 API Gateway 有什么区别？

**参考答案：**

| 对比项 | Agent Router | API Gateway |
|--------|-------------|-------------|
| 路由依据 | 语义、意图、上下文、任务类型 | URL、Header、参数 |
| 面向对象 | Agent、模型、工具、任务链 | 后端服务 |
| 是否需要理解语义 | 通常需要 | 通常不需要 |
| 决策复杂度 | 较高 | 较低 |
| 使用场景 | AI Agent 系统、多模型调度 | 微服务、接口转发 |
